# Migrate dependency/build tooling from Poetry to uv

We decided to replace Poetry with `uv` for dependency management, virtualenv creation, and the pre-commit/CI/Dockerfile tooling that drives them, rather than keep Poetry, because `uv` is materially faster (single Rust binary, no separate resolver/venv/build-backend indirection) and simpler to wire into CI/Docker layer caching. This repo has no complex Poetry-specific packaging needs (`packages = [{include = "octopus_monitoring"}]` in `pyproject.toml` doesn't even match the actual `app/` layout), so there was no offsetting benefit to staying on Poetry.

## Considered Options

- **Stay on Poetry**: works today, but slower installs/CI, and the `poetry-check`/`poetry-lock`/`poetry-export` pre-commit hooks add friction the migration removes.
- **pip-tools**: more manual (separate `pip-compile`/`pip-sync` steps), no single-binary story, rejected as a lateral move with no real upside over `uv`.

## Consequences

- `poetry.lock` is replaced by `uv.lock`; anyone with a local Poetry venv must switch to `uv sync`.
- `requirements.txt` (previously regenerated by the `poetry-export` pre-commit hook for Dependabot's benefit) is dropped — `.github/dependabot.yml`'s `pip` ecosystem entry is replaced with `uv`, which reads `uv.lock` directly.
- `Dockerfile`'s multi-stage Poetry-venv dance is replaced with `uv`'s own install flow.
