# Composite primary key for `daily_consumption_summary`

Every existing table (`agreement`, `product_rate`, `consumption`) uses a single synthetic string `id` as its primary key, even where a natural key already exists, because their natural keys aren't guaranteed unique on their own (e.g. re-agreeing the same tariff). `daily_consumption_summary` is different: `(energy, date)` is inherently unique by construction — one row is written per energy per calendar day, never more — so a synthetic `id` would just duplicate the natural key with no benefit. We use a true composite primary key `(energy, date)` instead, the first table in this schema to do so. This is a deliberate deviation from the rest of the schema's shape, not an oversight; don't "fix" it to match the synthetic-id pattern.
